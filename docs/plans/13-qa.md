## 1. Тест-стратегия и пирамида

Цель — максимизировать скорость feedback-loop разработчика и гарантировать стабильную ежедневную доставку digest'а.

```
           /\
          /  \  5%  E2E (staging VL + dev TG bot + CLI once)
         /----\
        /      \ 25% Integration (testcontainers VL + httptest TG)
       /--------\
      /          \ 70% Unit (pure Go, fake interfaces, table-driven, fuzz)
     /------------\
```

### 1.1. Уровень Unit (≈ 70% тестов, ≥ 75% coverage для MVP)

**Что покрываем:** вся чистая логика без I/O — dedup-нормализация, рендер шаблонов, парсинг LogsQL-ответа (JSON-lines декодер), конструктор Grafana-URL, загрузка/валидация конфига, маскирование секретов, builder cover-сообщения, idempotency-key и работа с time-windows (MSK↔UTC).

**Стиль:** table-driven + `t.Parallel()` + `t.Cleanup()` для временных ресурсов + `require` для инвариантов / `assert` для финальных чеков (подтверждено Context7 `/stretchr/testify`).

**Mocks:** интерфейсы `VLClient`, `TGClient`, `StateStore`, `Clock` — либо hand-written fakes, либо `go.uber.org/mock` (gomock, актуальная версия через `mockgen -typed`). Предпочтение — hand-written fakes для простых случаев, gomock — только для сложных ордер-зависимых expectation.

### 1.2. Уровень Integration (≈ 25% тестов)

**Что покрываем:** связки модулей против реальной или реально-подобной инфраструктуры в процессе CI:
- `collector` ↔ реальная VictoriaLogs (testcontainers-go, образ `victoriametrics/victoria-logs:latest`, wait-strategy `ForHTTP("/health")`);
- `delivery` ↔ мок TG (`net/http/httptest.NewServer` с программируемыми ответами: 200 / 429 / 5xx);
- `state` ↔ реальный SQLite-файл на `t.TempDir()`, включая WAL-режим, concurrent reads, file lock.

**Ожидаемое время всего integration-набора:** ≤ 3 мин wall-clock (testcontainers со стартом VL — ~8с per suite).

### 1.3. Уровень E2E (≈ 5% тестов, CI только на релизных тэгах)

**Что покрываем:** один happy-path прогон бинаря `log_analyser once --date=YYYY-MM-DD` в staging (реальная VictoriaLogs read-replica, dev TG bot с тест-чатом). Проверяем:
- cover-сообщение доставлено (`getMessage` по `message_id` из state);
- 5 attachments с правильными именами;
- Grafana-deeplinks парсятся (regex + URL-parse) и дают окно ±60с к тому, что указано в файле;
- в state появилась запись `runs` со статусом `delivered`.

**Гейтинг:** E2E не блокирует PR — только merge в `master` и релизные ветки (`release/v*`).

## 2. Критерии приёмки (DoD)

Каждому FR — ≥ 1 AC. Расширяет DoD из issue (D1..D5) и раздела 14 analysis.md. Формат: `AC-<FR>-<N>`.

### FR-1. Ежедневно 08:00 MSK за окно `[now−24h, now)`
- **AC-1-1.** При старте daemon в 07:59:59 MSK первый digest cycle запускается в ≤ 08:00:01 MSK (tolerance ±1s от cron trigger).
- **AC-1-2.** Окно выборки — ровно `[YYYY-MM-DD 08:00:00 MSK − 24h, YYYY-MM-DD 08:00:00 MSK)`. Проверяется через фейковые часы (`Clock` interface).
- **AC-1-3.** При прогоне 2026-03-27 (фиктивный DST, которого в РФ нет) сервис обрабатывает `Europe/Moscow` без смещений — тест покрывает TZDB edge-case.

### FR-2. Только level in {error, critical} на 5 хостов
- **AC-2-1.** Запрос в VL содержит фильтр `(level:=error OR level:=critical)` и стрим-селектор по всем хостам из конфига.
- **AC-2-2.** Записи с `level=warn/info/debug` отфильтровываются на уровне LogsQL (не в Go), проверка — захваченный request в testcontainers VL.
- **AC-2-3.** При пустом `HOSTS` daemon отказывается стартовать с exit-code ≥ 2 и сообщением в stderr «at least one host must be configured».

### FR-3. Long-running daemon + инкрементальный state
- **AC-3-1.** Daemon проходит 3 подряд digest cycle'а без утечек goroutine (`goleak` check в teardown).
- **AC-3-2.** Между запусками state-файл обновляется: `runs.last_completed_at` растёт монотонно.
- **AC-3-3.** `SIGTERM` → graceful shutdown ≤ 30с, в логах `shutdown complete`, незаконченные cycle отмечены `interrupted`.

### FR-4. Отдельный файл на каждый из 5 хостов
- **AC-4-1.** Имена строго `<host>_YYYY-MM-DD.md`, допустимые символы в host — `[a-z0-9-]`. Любой host с запрещённым символом → fail-fast на старте.
- **AC-4-2.** При отсутствии ошибок для хоста файл всё равно формируется с телом «За окно ошибок не обнаружено» (см. T-2).
- **AC-4-3.** Файлы сохраняются в `reports/YYYY-MM-DD/` и retention чистит их через N дней (по умолчанию 30, FR-18).

### FR-5. Содержимое файла
- **AC-5-1.** Каждая из шести обязательных секций присутствует (заголовок / агрегаты по app / топ-N инцидентов / паттерны / прочее / deeplinks). Snapshot-тест (см. §9).
- **AC-5-2.** Ровно один deeplink в заголовке + по одному на каждый инцидент.
- **AC-5-3.** `<_msg>` в выдержках никогда не превышает 2 KB (truncate с ellipsis).

### FR-6. Cover-сообщение
- **AC-6-1.** `sendMessage.text` ≤ 4096 символов (жёсткая проверка длины, fail-fast при превышении).
- **AC-6-2.** `parse_mode=HTML`, содержит `<pre>` с таблицей host×count + одну `<a href>` на общую Grafana-панель.
- **AC-6-3.** Формат таблицы проходит snapshot-тест.

### FR-7. sendMediaGroup с fallback
- **AC-7-1.** Нормальный путь: 1 `sendMessage` (cover) + 1 `sendMediaGroup` (5 `InputMediaDocument`).
- **AC-7-2.** Mock TG возвращает 400 `Bad Request: media group can't be sent` → fallback в 5 последовательных `sendDocument`, тест фиксирует 5 запросов.
- **AC-7-3.** Идемпотентность: повторный send с тем же `run_id` → skip, метка `already delivered`.

### FR-8. Grafana-deeplinks точность ±60с
- **AC-8-1.** URL парсится, `range.from/to` извлекаются, diff с окном в файле ≤ 60000ms.
- **AC-8-2.** `expr` в query содержит escaped-версию fingerprint-шаблона; URL-decode → канонический LogsQL.
- **AC-8-3.** E2E HEAD-запрос на Grafana-URL возвращает 200 (в staging).

### FR-9. Конфиг через env
- **AC-9-1.** Все 12 переменных (TG_BOT_TOKEN, TG_CHAT_ID, VL_URL, GRAFANA_URL, GRAFANA_ORG_ID, GRAFANA_VL_DS_UID, HOSTS, SCHEDULE_CRON, TZ, LEVELS, NOISE_K, TOP_N) обязательны (кроме NOISE_K/TOP_N/LEVELS с дефолтами).
- **AC-9-2.** Отсутствие обязательной переменной → exit 2 + полный список missing в stderr (не разбивать на N последовательных сообщений).
- **AC-9-3.** Дамп config при старте **не содержит** значения `TG_BOT_TOKEN` и VL basic-auth password.

### FR-10. Dedup fingerprint
- **AC-10-1.** 1000 строк с одним шаблоном и разными trade-IDs → 1 fingerprint, `count=1000`.
- **AC-10-2.** Разные `host` с одним шаблоном → разные fingerprints (host — часть хэша).
- **AC-10-3.** Fuzz-инвариант: `normalize(normalize(x)) == normalize(x)` (см. §8).

### FR-11. Шум-порог
- **AC-11-1.** Инциденты с `count < NOISE_K` (default 3) в основной секции отсутствуют.
- **AC-11-2.** Блок «Прочее» показывает `N инцидентов, M записей`, суммы совпадают с исходными.

### FR-12. Идемпотентность повторного запуска
- **AC-12-1.** Повторный `once --date=2026-04-22` после успешной доставки → exit 0, в логах `skip: already delivered run_id=<...>`, в TG ничего не отправлено (captured по mock-счётчику).

### FR-13 (v0.2). Realtime alerts
- **AC-13-1.** Новый fingerprint с `level=critical`, не встречавшийся 7 дней → отправляется в отдельный thread ≤ 60с.
- **AC-13-2.** Throttle: 1 alert per fingerprint per 10 мин.

### FR-14 (v0.2). Proxy
- **AC-14-1.** SOCKS5 и HTTP(S) прокси работают для TG-трафика; при недоступности прокси — graceful fail + dead-letter.

### FR-15 (v0.3). UI read-only
- **AC-15-1.** UI показывает инциденты за 30 дней, фильтры по host/app/level работают.

### FR-16. Cover-агрегаты «тип X / других Y»
- **AC-16-1.** В cover для каждого host: топ-1 pattern с counts + `others: N`. Сумма сходится.

### FR-17 (v0.2 опц.). LLM-резюме
- **AC-17-1.** При `LLM_ENABLED=false` секция отсутствует в файле, отчёт уходит.
- **AC-17-2.** При LLM 5xx / timeout > 10s — секция пропускается, остальной отчёт доставляется.

### FR-18. Retention отчётов
- **AC-18-1.** Файлы старше `RETENTION_DAYS` удаляются; метрика `reports_deleted_total` инкрементится.

### FR-19. CLI-подкоманды
- **AC-19-1.** `run` — запускает daemon.
- **AC-19-2.** `once --date=YYYY-MM-DD` — разовый прогон за сутки; не затирает slot ежедневки.
- **AC-19-3.** `health` — проверяет `VL:/health` + `TG:getMe` + SQLite open; exit 0 / 1.
- **AC-19-4.** `version` — выводит SemVer из VERSION файла + commit hash.

## 4. Unit-тесты по модулям

### 4.1. `internal/collector`

**Тестируем:**
- Построение LogsQL-запроса: правильная конкатенация `{host=...}`, `_time:[...)`, `level:=...`.
- URL-encoding параметров.
- Парсинг stream JSON-lines (включая битые строки — skip + counter).
- Применение retry с экспоненциальным backoff (через инъекцию `Clock`).

**Фикстуры:** `tests/fixtures/logs/*.jsonl` (см. §7).

**Mocks:** `httptest.NewServer` с программируемым ответом (JSON-lines из файла).

**Имена функций:**
- `TestCollector_BuildLogsQL_SingleHost`
- `TestCollector_BuildLogsQL_TimeWindow_MSK`

- `TestCollector_Retry_ExponentialBackoff`
- `TestCollector_StreamingMemoryBound` (проверяет, что вся выдача не буферизуется)

**Coverage target:** ≥ 85%.

## 9. Snapshot-тесты рендера

// pseudo, спецификация для executor'а
want := mustReadFile("testdata/snapshots/t5_2026-04-22.md")

## 18. Отличия от аналитики

**Что расширяется / уточняется относительно `00-analysis.md` §13.4:**

1. **T-1..T-12 → T-1..T-30.** Добавлены: отдельные тесты нормализации (T-13..T-17), шум-порог (T-18), security (T-19..T-20), SQLite-WAL concurrency (T-21), file-lock (T-22), per-host isolation (T-23), stream memory bound (T-24), dedup retention (T-25), observability (T-26..T-27), dead-letter recovery (T-28), fuzz (T-29), snapshot (T-30).

2. **Возражение к dedup-профилю из FR-10:** regex-based normalize — хрупкий. **Предлагаю** вынести regex-профиль в yaml-файл (`config/dedup-profile.yml`), чтобы ops мог менять без редеплоя. Для MVP — да, hardcoded, но в v0.2 — обязательно в конфиг. Не менять — значит каждый новый «паттерн для trade-ID» потребует нового релиза.

3. **Возражение к E2E-объёму:** analysis.md перечисляет «Grafana-deeplink: клик → открывается Explore» как T-9 — это ручной тест. В §6 я делаю автотест через HEAD + URL-parse. Клик-проверка остаётся ручной, но happy-path deeplink-валидация автоматизирована.

4. **Добавлено явно:** property-based тесты (в analysis.md отсутствуют). Инвариант `normalize(normalize(x))=normalize(x)` нельзя надёжно проверить table-driven'ом — нужен fuzz.

5. **Bug-policy / SLA** — в analysis.md нет (не требовалось от аналитика, но обязательно для QA).
