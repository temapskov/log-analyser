# ADR-0001: Фиксировать архитектурные решения в ADR

- **Статус:** Accepted
- **Дата:** 2026-04-23
- **Автор:** temapskov
- **Контекст:** инициализация проекта `log_analyser` (issue [QCoreTech/awesome#908](https://github.com/QCoreTech/awesome/issues/908)).

## Контекст

Проект `log_analyser` стартует с нескольких одновременно обсуждаемых архитектурных развилок (state-store, формат отчёта, клиент Telegram, планировщик, templating, логгер, прокси, модульность). Обсуждения будут идти между аналитиком, архитектором, SRE, DevOps, QA и TeamLead. Нужен легко-читаемый артефакт, по которому через 6 месяцев новый член команды поймёт **почему** выбрали именно так.

## Решение

Все значимые архитектурные и процессные решения фиксировать как **Architecture Decision Records** (ADR) в каталоге `docs/adr/` по шаблону Майкла Нигарда.

### Правила

1. Имя файла: `NNNN-short-kebab-title.md`, где `NNNN` — монотонно возрастающий 4-значный номер.
2. Обязательные секции каждой ADR: **Статус**, **Дата**, **Автор**, **Контекст**, **Решение**, **Последствия** (обязательно указывать минусы и trade-offs), **Альтернативы** (перечислить и объяснить, почему отвергнуты).
3. Статусы: `Proposed → Accepted → Deprecated → Superseded by ADR-XXXX`. Не правим «историю»: замену оформляем новой ADR и проставляем `Superseded by`.
4. Решение считается принятым, когда PR с ADR `Accepted` смерджен в `master`.
5. Любой код, реализующий ADR, ссылается на её номер в коммите или комментарии (только в non-obvious местах).

### Жизненный цикл

- `Proposed` — ADR описывает предлагаемое решение, идёт обсуждение в PR/issue.
- `Accepted` — решение принято, команда следует ему.
- `Deprecated` — решение больше не рекомендуется, но ещё живо в коде.
- `Superseded by ADR-XXXX` — решение заменено другим ADR.

## Последствия

**Плюсы**
- Единое место истины для «почему так».
- Низкий порог входа для новых разработчиков.
- ADR'ы легко диффать в PR и обсуждать.

**Минусы**
- Требует дисциплины: не забывать оформлять ADR **до** или **одновременно** с реализацией.
- Для тривиальных решений ADR — лишний шум (мы договариваемся: только архитектурные/межмодульные вопросы).

## Альтернативы

- **Писать всё в Wiki / Confluence.** Отвергнуто: внутренний корпоративный Confluence не всегда доступен, и файлы в репозитории живут вместе с кодом.
- **Ничего не фиксировать, держать в головах.** Отвергнуто: через 3 месяца никто не вспомнит, почему `text/template`, а не `templ`.
- **Дизайн-доки (RFC) в одном большом файле.** Отвергнуто: плохо масштабируется, не видно истории изменений.
